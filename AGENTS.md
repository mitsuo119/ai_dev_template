# Project Guidelines

AI駆動開発のためのプロジェクトガイドライン。
Karpathy の4原則 + Superpowers の方法論をベースに構築。

## Core Principles (Karpathy-Inspired)

### 1. Think Before Coding

- 仮定を明示的に述べること。不確実なら確認する
- 複数の解釈がある場合、すべて提示する。暗黙的に選択しない
- よりシンプルなアプローチがあれば提案し、必要に応じて反論する
- 不明確な点があれば止まり、何が不明確かを名指しして確認する

### 2. Simplicity First

- 要求されたこと以上の機能を追加しない
- 一度しか使わないコードに抽象化を適用しない
- 要求されていない「柔軟性」や「設定可能性」を追加しない
- 起こり得ないシナリオのエラーハンドリングを書かない
- 200行で書けるものが50行で済むなら、書き直す
- 判断基準: シニアエンジニアが「複雑すぎる」と言うなら、簡素化する

### 3. Surgical Changes

- 隣接するコード、コメント、フォーマットを「改善」しない
- 壊れていないものをリファクタリングしない
- 既存のスタイルに合わせる（自分のやり方と違っても）
- 無関係なデッドコードに気づいたら削除せず言及する
- 自分の変更で不要になったインポート/変数/関数は削除する
- テスト: 変更した全行がユーザーのリクエストに直接つながるか

### 4. Goal-Driven Execution

- タスクを検証可能なゴールに変換する
- 「バリデーション追加」→「無効な入力のテストを書き、パスさせる」
- 「バグ修正」→「再現するテストを書き、パスさせる」
- 「リファクタリング」→「前後でテストがパスすることを確認」
- 複数ステップのタスクは簡潔な計画を立てる:
  ```
  1. [ステップ] → 検証: [チェック方法]
  2. [ステップ] → 検証: [チェック方法]
  ```

## Development Workflow

### Feature Development

1. **ブレスト**: コードを書く前に要件を整理し、設計を議論する
2. **計画**: 実装計画を2-5分のタスクに分割する
3. **TDD**: RED → GREEN → REFACTOR サイクルを遵守する
4. **レビュー**: 計画に対してコードをレビューする
5. **完了**: テスト通過を確認し、ブランチを仕上げる

### Bug Fixing

1. 再現手順を確認する
2. 再現するテストを書く（RED）
3. 最小限の修正を行う（GREEN）
4. リグレッションがないことを確認する

### Code Review

- セキュリティ問題は即座にブロック
- パフォーマンス影響を定量的に評価
- テストカバレッジを確認
- OWASP Top 10 に照らした脆弱性チェック

## Code Style

- コードは英語で記述する（変数名、関数名、クラス名など）
- コメントおよびドキュメンテーションは日本語で記述する
- 命名は意図を明確にする（`data` → `userProfiles`, `tmp` → `unprocessedItems`）
- マジックナンバーは定数化する
- 関数は単一責任を守る
- ネストは最大3レベルまで（早期リターンで回避）

### Python
- [Google Python Style Guide](https://google.github.io/styleguide/pyguide.html) に準拠する
- docstring は Google スタイルで記述する（日本語）
- 型ヒントをすべての関数に付ける

## Git Conventions

- コミットメッセージは [Conventional Commits](https://www.conventionalcommits.org/) に従う
  - プレフィックスは英語: `feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`
  - 本文は日本語で記述する
  - 例: `feat: ユーザー認証機能を追加`
- ブランチ名: `feat/機能名`, `fix/バグ名`, `docs/ドキュメント名`
- 1コミット = 1論理的変更
- WIPコミットはスカッシュしてからマージ

## Security

- シークレットをコードにハードコードしない（環境変数を使う）
- ユーザー入力は必ずバリデーションする
- SQLインジェクション、XSS、CSRFを防止する
- 依存関係は定期的にアップデートする
- `.env` ファイルは `.gitignore` に含める
